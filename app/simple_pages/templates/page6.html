CI and CD stand for continuous integration and continuous delivery/continuous deployment. Continuous Integration (CI) is a development practice where developers integrate code into a shared repository frequently, preferably several times a day. In continuous development updates are made continuously, piece-by-piece, enabling software code to be delivered to customers as soon as it is completed and tested.


GitHub Actions

Github actions help to automate, customize, and execute software development workflows right in your repository. We can discover, create, and share actions to perform any job we'd like, including CI/CD, and combine actions in a completely customized workflow. This project uses features from github actions to implement CI/CD.

In the project we have two ".yml" files which are inside our ".github/workflows" directory. This is used for CI/CD purposes and the files contain routines to follow on certain actions. These are executed under github ActionsMore about these will be explained below.

How CI/CD is Used in the Project

"Integration is primarily about communication. Integration allows developers to tell other developers about the changes they have made. Frequent communication allows people to know quickly as changes develop."[1]

To implement Integration in the project we have a repository on Github for this project. The repository is made public. The repository has a main branch and a development branch. Changes are made to the development branch and comitted to the same branch. A pull request is then created to merge the branch.

"Continuous Delivery is a software development discipline where you build software in such a way that the software can be released to production at any time."[2]

Continuous delivery has also been set up for this project. As mentioned in the previous section we have "dev.yml" and "prod.yml" that defines the workflows for our repository. These workflows contain actions/jobs to be executed on certain criteria defined by us. On a push or merge made to the main branch the "prod.yml" workflow will run. This workflow runs pytest, and deploys the project onto heroku(production-site) and docker.

On pull requests "dev.yml" workflow is executed which runs pytest and deploys the project to a development-site and not in the production-site in heroku. After all the tests have passed we can merge the branch the main branch. As we can see these all automate the deployment and testing for the project. This is how CI/CD is used in the project.

How Image is Deployed in Dockerhub

The workflows mentioned above deploys the image onto DockerHub on push/merge into the master branch as defined in our "prod.yml" file. The changes in the docker image is commited and pushed into the docker repository. This happens automatically through github action workflows.
